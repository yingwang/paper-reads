## I. The Problem

A visual agent is a system that decides its next action by watching a screen or a game frame: the browsing agent inside browser-automation stacks, the OS-level desktop agent, the exploration agent in games like Minecraft all fall into this category. One of the dominant lines of progress for such agents over the past few years has been to externalise reusable skills into a repository, so that when a new task arrives the model first looks up ready-made procedural knowledge instead of rediscovering it from scratch each time. The idea works well on text-only tasks, but its limitations surface as soon as it is ported to visual tasks.

What do text skill repositories actually store? In most cases procedural knowledge is compressed into a natural-language prompt, a short executable script, or a policy learned from demonstrations. All three formats share the same blind spot: they describe only "what to do" and not "what the screen should look like when you do it" or "what the screen should look like once it is done". A visual agent operates precisely at that layer. The local state of a screen is not a tuple of discrete labels but a stream of continuous pixel evidence, and judging whether an action took effect requires observing whether a button has changed colour, whether a dialog has popped up, whether a progress bar has advanced. A text skill neither points the agent at the right region of the screen nor supplies a reference image to compare against, so the agent either fails to recognise when the skill applies, or applies it in the wrong place.

The authors accordingly introduce the notion of multimodal procedural knowledge: a complete reusable skill must describe not just the operation but the visual evidence for the triggering condition, the visual evidence for progress, and the visual evidence for completion or failure. Three concrete questions follow. First, the representation question: what fields should a multimodal skill package contain. Second, the generation question: how to automatically mine such skills from publicly available interaction trajectories rather than relying on hand authoring. Third, the usage question: how to deliver multimodal evidence to the agent at inference time without flooding the context with reference screenshots and without letting those references anchor the agent so strongly that it ignores the actual current frame.

## II. Method

The method has three parts: the structure of the skill package, the generator that distils public trajectories into skills, and the branch-loading mechanism used at inference time.

The structure of the skill package. Each MMSkill is a four-tuple M = (D, P, S, K). D is a short descriptor used by the upper-level system for retrieval. P is a reusable program written in text, effectively a step-by-step instruction sheet. S is a set of runtime state cards, each describing how to handle one state: which cues in the current frame confirm that state, when the skill should and should not be applied in that state, how to verify completion of the operation, and which views of the frame are worth inspecting. K corresponds one-to-one with S: each entry is a keyframe bundle containing several images from different viewpoints. Common viewpoints include full_frame (preserves global context), focus_crop (a local zoom aligned to the target widget), before (frame before the operation), and after (frame after). One sentence from the paper captures the structure: the program states the reusable flow, the state cards state when that flow is applicable, and the keyframes let the state be recognised at runtime by a human or by a vision model.

The skill generator is a five-stage pipeline driven entirely by LLMs, whose job is to mine a skill library from public non-evaluation trajectories, that is, the operation recordings and logs already in public data. Stage zero is task embedding and clustering: each trajectory's task description and metadata are embedded into a vector space, then grouped into semantically focused clusters, so that "open the browser and book a flight" and "filter rows in a spreadsheet" land in different buckets. Stage one is skill planning: an LLM agent proposes a set of atomic skills for each cluster, writes out the boundary of each, and outputs a domain plan. Stage two is skill merging: skills that are duplicated across clusters are merged, and skills that are too broad, that act like umbrellas covering unrelated operations, are rejected, leaving a refined set of skill specifications. Stage three is text-first drafting: the model does not look at images yet but picks several reference tasks and drafts the descriptor, the reusable program, and the planned state cards in text. Stage four is image grounding and audit: the chosen keyframes are then read, each focus_crop is anchored to the specific UI region it refers to, multi-view keyframe bundles are assembled, and a set of reusable meta-skills runs a quality check over the whole package. The core idea of the pipeline is to not store raw demonstrations as skills, but to compress them into a compact piece of visual procedural knowledge. This saves storage and context, and also prevents idiosyncrasies in the demonstrations from contaminating the skill.

The way skills are consumed at inference time is called the branch-loaded multimodal skills agent. Dumping a full skill package along with all of its keyframes into the main agent's context is a poor choice, for three reasons: context pressure rises sharply; reference images and the current frame appear simultaneously, and the model is easily confused about which one to attend to; a visual anchoring effect emerges in which the model plans actions around the reference image rather than around the current frame. Branch loading splits the process in two. The first stage is gated view selection: in a temporary branch, given the current observation O_t, the history H, the skill program P, and the set of state cards S, the system decides which state cards J_t and which views R_t to load. Only the selected images ever appear in the main context; unselected ones do not. The second stage is branch planning: the temporary branch consumes the selected state cards and views and outputs a structured directive G_t = (applicability, sub-goal, plan, things-not-to-do, verification). By the time this directive returns to the main agent it no longer carries the raw screenshots or the verbose text; only the distilled decision support remains, which severs the visual contest between reference and current frame. The executable action is still issued by the main agent against the live frame.

Why does this design address the pain points raised in the previous section. State cards solve "when to apply"; keyframe bundles solve "does the frame look right"; branch loading solves "the main context is not crowded out by reference images and the planning is not led astray by them". Each component covers one stage, and together they form a closed loop from state recognition to action execution.

## III. Experiments

The evaluation sits on four visual agent benchmarks. OSWorld contributes 360 test tasks against an Ubuntu desktop, covering the browser, LibreOffice, creative tools, system settings, and code editors. macOSWorld contributes 143 tasks, swapping the backdrop to macOS and covering file management, media, productivity, and system tasks. VAB-Minecraft is a Minecraft subset of Visual Agent Bench, with tasks for item acquisition, recipe inference, and tool use. Super Mario Bros comes from LMGame-Bench, where the goal is to clear game levels. The model lineup spans frontier models such as Gemini 3.1 Pro, Gemini 3 Flash, Qwen3-VL-235B, and Kimi-K2.6, as well as smaller models like GLM-5V and Qwen3-VL-8B-Instruct.

Headline results (Tables 1 and 2 of the paper) are reported per benchmark and per model. On OSWorld, Gemini 3.1 Pro rises from 44.08% to 50.11% (+6.03 points), Gemini 3 Flash from 36.65% to 47.97% (+11.32 points), Qwen3-VL-235B from 21.34% to 39.17% (+17.83 points), and Qwen3-VL-8B-Instruct from 10.78% to 25.40% (+14.62 points). On macOSWorld, Gemini 3 Flash rises from 55.94% to 65.73% (+9.79 points) and GLM-5V from 34.97% to 51.75% (+16.78 points). On VAB-Minecraft, Gemini 3 Flash success rate rises from 67.24% to 73.28% (+6.04 points) and Qwen3-VL-235B from 52.59% to 62.07% (+9.48 points). One consistent trend is that the weaker the base model, the larger the gain from attaching external skills. This matches the authors' claim that external multimodal procedural knowledge complements the model's built-in priors: the fewer priors a model has, the more room external procedural knowledge has to fill.

The ablations (Figure 3) yield two findings worth noting. The first is on skill-package content: removing the state cards measurably degrades the agent's judgement of whether the current state is one in which the skill applies; removing the keyframe images strips away the visual anchoring evidence, and operational precision drops. Each handles one role and they do not substitute for each other. The second is on the branch-loading mechanism itself: pouring the entire skill package into the main context (direct-full loading) actually drags performance down; view selection without branch planning is the second best; full two-stage branch loading is the best. The engineering takeaway is that letting any extra information into the main context of an LLM agent carries a real cost, and evidence filtering plus formal compression are not optional.

Two sets of behavioural numbers (Table 3 and Figure 4) are also interesting. The first is skill-invocation rate: on OSWorld, Gemini 3 Flash's invocation rate rises from 41.11% with text-only skills to 62.50% with MMSkills, and Qwen3-VL-235B from 37.50% to 65.28%. The reading is that once visual evidence is attached, the agent finds it easier to recognise "this is a state in which the skill applies" and is more willing to invoke it. The second is trajectory efficiency: Gemini 3 Flash's average step count drops from 13.11 to 11.86 (1.25 steps fewer), and Qwen3-VL-235B's from 15.22 to 9.87 (5.35 steps fewer). In other words, MMSkills does not pile additional advisory actions onto the flow; it actually shortens the entire trajectory. The third is the shift in action distribution: Qwen3-VL-235B's share of clicks on OSWorld drops from 75.8% to 63.7%, while the share of keyboard actions and DONE (completion confirmation) rises, and the share of consecutive repetitions of the same action drops from 21.8% to 6.2%. It reads like a transition from "flailing" to "look at the frame and then act".

The viewpoint-selection statistics expose another piece of engineering wisdom: focus_crop is the most frequently selected viewpoint (241 out of 352 view selections by Gemini 3 Flash on OSWorld land on focus_crop), full_frame is chosen only when global context is needed, and before/after appear least often, used for comparative evidence of progress or completion. The practical implication for deployment is that if resources are scarce, invest first in making focus_crop solid.

## IV. Limitations

The authors list four. First, the coverage of the skill repository is bounded by the coverage of the source trajectories: in domains untouched by public data, the generator produces no corresponding skills. Second, both skill generation and visual grounding rely on the LLM, and bad labels or misplaced focus regions propagate downstream. Third, branch loading itself has inference cost: it requires an extra gating and planning pass, increasing the number of model calls and the latency, and the paper does not report end-to-end latency numbers. Fourth, applying this idea to safety-sensitive scenarios would require stronger verification and online skill-repair mechanisms, which the current version does not provide.

A few additional questions are worth flagging after the read. First, the evaluation is confined to GUI operations and lightweight games; whether the approach transfers to real-world visual tasks with more complex scenes and trajectories, such as robot manipulation, driving assistance, or medical imaging support, has no empirical backing here. Second, the maintainability of the skill library: the paper shows a one-shot distilled library, but long-term evolution will run into familiar problems such as UI redesigns and operating-system updates, and it is unclear whether old keyframes will still be recognised under a new interface or whether the "visible cues" in a state card will survive a repaint. Third, the coupling to the underlying LLM: both the generator and the gating depend on the base model's image-text understanding, and swapping the base model may require a full re-evaluation. Fourth, while the relative gains on small models look large, the baselines are also low (Qwen3-VL-8B going from 10.78% to 25.40% means absolute success is still under one-third), which is far from the regime where the agent can be left to run on its own.

## One Sentence

Extending reusable skills from plain text into a multimodal package of program plus state cards plus multi-view keyframes, and using branch loading to break the visual contest between reference images and the live frame, produces consistent gains across four visual-agent benchmarks.
