## 一、问题

视觉智能体（visual agent）是一类需要通过观察屏幕画面或游戏画面来决定下一步动作的系统：浏览器自动化里的浏览代理、操作系统层面的桌面智能体、Minecraft 这类游戏里的探索智能体，都属于此列。过去几年这类智能体能力提升的一条主要路线，是把"可复用的技能"沉淀为一个外部仓库，让模型在新任务到来时先去仓库里取用现成的过程性知识，而不是每次都从零摸索。这思路在纯文本任务上行之有效，可一旦搬到视觉任务上便露出了不足。

文本技能仓库存的是什么。多数情况是把过程性知识压缩成自然语言提示，或者是几段可执行的脚本，或者是一段由演示学到的策略。这三种形式共有的盲点是：它们只描述"该做什么"，却不描述"画面长什么样时该做"和"做完后画面会变成什么样"。视觉智能体面对的恰恰是这一层：屏幕的局部状态不是一组离散标签，而是连续的像素证据；判断一个操作是否生效，需要看按钮颜色有没有变、对话框有没有弹出、进度条有没有推进。文本技能既不告诉智能体应当去看画面的哪一处，也不给出"对照画面"用以比对，结果是该用的时候没认出来，认出来也用错了地方。

作者据此提出一个叫多模态过程性知识（multimodal procedural knowledge）的概念：一份完整的可复用技能不能只描述操作，还得描述触发条件的视觉证据、过程进展的视觉证据、完成或失败的视觉证据。围绕这一概念，他们要回答三个具体问题。其一是表示问题，即一份多模态技能包到底该包含哪些字段。其二是生成问题，即如何从公开的交互轨迹里自动把这种技能挖出来，而不是依赖人手编写。其三是使用问题，即推理时如何把多模态证据交到智能体手上，而又不至于让大批参考截图挤占上下文、也不至于让智能体被参考图反过来锚住、忽略实际画面。

## 二、方法

整个方法分成三块：技能包的结构、把公开轨迹炼成技能的生成器、推理时的分支加载机制。

技能包的结构。每个 MMSkill 是一个四元组 M = (D, P, S, K)。D 是简短描述符，让上层去检索这个技能。P 是文本写就的可复用程序，相当于"步骤说明书"。S 是若干张运行时状态卡（state card），每张记录一种状态下的应对：当前画面有哪些线索能确认此状态、该状态下何时该用此技能何时不该用、操作完成的验证方式、需要看画面的哪几个视图。K 与 S 一一对应，是若干个关键帧束（keyframe bundle），每束含多个视角的图像，常见的视角有 full_frame（保留全局上下文）、focus_crop（局部放大、对齐到目标控件）、before（操作前）与 after（操作后）。一句论文原话可以总结这一结构：程序说明可复用流程，状态卡说明流程何时有效何时无效，关键帧让状态在运行时被肉眼或视觉模型识别出来。

技能生成器是一条五阶段的流水线，全部由 LLM 驱动，作用是从公开的非评估轨迹（即公开数据里那些操作录像与日志）中提炼出技能库。第零阶段是任务嵌入与聚类：把每条轨迹的任务说明与元数据嵌入向量空间，再聚成若干语义聚焦的簇，相当于把"打开浏览器订机票"和"在电子表格里筛行"分到不同的桶里。第一阶段是技能规划：一个 LLM 代理对每个簇提议若干原子技能，写明每个技能的边界，输出"领域规划表"。第二阶段是技能合并：把不同簇里重复的技能合并掉，把过于宽泛、像伞一样把不相干操作罩在一起的技能拒绝掉，得到一份精炼的技能规范集合。第三阶段是文本优先起草：不读图，先选好几条参考任务，把描述符、可复用程序、计划中的状态卡先用文字写好。第四阶段是图像锚定与审计：再回头去读所选关键帧，定位每个 focus_crop 所对应的具体 UI 区域，构造多视角的关键帧束，最后用一组可复用的元技能（meta-skill）对整份技能包做一次质检。这一流水线的核心立意是不把原始演示当成技能本身存进库里，而是把它压缩成"紧致的视觉过程性知识"。这件事既能节省存储与上下文，也避免了演示数据里偶发的特异性把技能污染掉。

推理时使用技能的方式叫分支加载多模态智能体（branch-loaded multimodal skills agent）。直接把整份技能包连同它的所有关键帧塞进主智能体的上下文不是好主意，会带来三种麻烦：一是上下文压力陡增；二是参考图与当前画面同时出现，模型容易混淆该看哪张；三是出现视觉锚定（visual anchoring）效应，模型围绕参考图里的样子规划动作，而不是围绕当前画面的实际情况。分支加载分两段。第一段叫门控视图选择：在一个临时分支里，先根据当前观察 O_t、历史 H、技能程序 P 与状态集合 S，决定要把哪几张状态卡 J_t 与哪几个视角 R_t 加载进来，主上下文里只会看见被选中的那几张图，未中选的不进。第二段叫分支规划：临时分支拿着所选状态卡与视图，输出一组结构化的指导 G_t = (是否适用、子目标、计划、不要做什么、验证方式）。这份结构化指导回到主智能体那里时，已经不带原始截图与详细说明文本了，只有提炼好的决策支持，参考图与当前画面的视觉对抗于是被切断。可执行的动作仍由主智能体在当前画面上完成。

这套设计为什么能解决"问题"那一节提到的痛点。状态卡解决了"何时该用"的问题，关键帧束解决了"画面看起来对不对"的问题，分支加载解决了"上下文不被参考图占满、规划不被参考图带偏"的问题。三件事各管一段，串起来形成一条从"识别状态"到"执行动作"的闭环。

## 三、实验

评测落在四个视觉智能体基准上。OSWorld 共 360 条测试任务，背景是 Ubuntu 桌面，覆盖浏览器、LibreOffice、创意工具、系统设置、代码编辑器等。macOSWorld 共 143 条，背景换成 macOS，覆盖文件管理、媒体、生产力、系统任务。VAB-Minecraft 是 Visual Agent Bench 里 Minecraft 的一段子集，任务是物品获取、配方推理、工具使用。Super Mario Bros 来自 LMGame-Bench，做游戏关卡通关。模型梯度上既有 Gemini 3.1 Pro、Gemini 3 Flash、Qwen3-VL-235B、Kimi-K2.6 这一类前沿模型，也有 GLM-5V 与 Qwen3-VL-8B-Instruct 这一类较小的模型。

主要结果（论文 Table 1 与 Table 2）按基准与模型分开报。OSWorld 上 Gemini 3.1 Pro 从 44.08% 升到 50.11%（+6.03 个百分点），Gemini 3 Flash 从 36.65% 升到 47.97%（+11.32 个百分点），Qwen3-VL-235B 从 21.34% 升到 39.17%（+17.83 个百分点），Qwen3-VL-8B-Instruct 从 10.78% 升到 25.40%（+14.62 个百分点）。macOSWorld 上 Gemini 3 Flash 从 55.94% 升到 65.73%（+9.79 个百分点），GLM-5V 从 34.97% 升到 51.75%（+16.78 个百分点）。VAB-Minecraft 上 Gemini 3 Flash 成功率从 67.24% 升到 73.28%（+6.04 个百分点），Qwen3-VL-235B 从 52.59% 升到 62.07%（+9.48 个百分点）。一个一致的趋势是，越是能力相对弱的模型，外接技能带来的增益越大。这与作者主张的"外部多模态过程性知识与模型内置先验互补"是吻合的：模型本身越缺先验，外挂的过程性知识能填的空间就越多。

消融实验（Figure 3）有两条结论值得记。其一是技能包内容的消融：拿掉状态卡，智能体在判断"现在是否处在该技能适用的状态"上明显变差；拿掉关键帧图像，智能体失去了视觉锚定证据，操作的精度下降。两者各管一段、互不替代。其二是分支加载机制的消融：把所有技能包内容一次性灌进主上下文的做法（direct-full loading）反而拖累表现；只做视图选择不分支规划，效果次之；完整的两段式分支加载效果最好。这指向一条工程性的判断：在 LLM 智能体里，让额外信息进入主上下文这件事本身就是有代价的，做证据筛选和形式压缩是有必要的。

行为侧的两组数字（Table 3 与 Figure 4）也颇有意思。其一是调用率：OSWorld 上 Gemini 3 Flash 的技能调用率从纯文本技能时的 41.11% 升到 MMSkills 的 62.50%，Qwen3-VL-235B 从 37.50% 升到 65.28%。这说明带了视觉证据之后，智能体更容易识别"现在是个适用该技能的状态"，于是更愿意去调用。其二是轨迹效率：Gemini 3 Flash 平均步数从 13.11 降到 11.86（少 1.25 步），Qwen3-VL-235B 从 15.22 降到 9.87（少 5.35 步）。换言之 MMSkills 不只是把更多咨询动作堆进流程里，反倒把整条轨迹缩短了。其三是行为分布的迁移：Qwen3-VL-235B 在 OSWorld 上的点击占比从 75.8% 降到 63.7%，键盘动作与 DONE（完成确认）的比例上升；连续重复同一动作的比例从 21.8% 降到 6.2%。读起来像是一个从"乱试一通"到"看清画面再动手"的转变。

视角选择的统计也透露了一点工程经验：focus_crop 是被选用最频繁的视角（OSWorld Gemini 3 Flash 上 352 次视图选择里有 241 次落在 focus_crop 上），full_frame 在需要全局上下文时才被选，before/after 出现频次最低、用于过程或完成的对比证据。这条经验对实际落地有参考意义：如果资源紧张，优先把 focus_crop 这一类视角做扎实。

## 四、局限

作者自己点到的有四条。第一，技能仓库的覆盖面受限于源轨迹的覆盖面：公开数据没碰过的领域，这套生成器也炼不出对应技能。第二，技能生成与视觉锚定都依赖 LLM，错误的标注或者错放的 focus 区域会传导到下游。第三，分支加载本身有推理成本，需要多走一遍门控与规划，模型调用次数与延迟都增加，论文没给出端到端的延迟数字。第四，要把这套思路用到安全敏感场景，还需要更强的验证机制与在线技能修复机制，目前的版本没做这一层。

读完之后还有几条可以补上的潜在问题。其一是评估范围都落在 GUI 操作与轻量游戏，对那些场景与轨迹更复杂的真实世界视觉任务（例如机器人操作、自动驾驶辅助、医疗影像辅助决策）能否迁移没有数据支撑。其二是技能库的可维护性。论文展示的是一次性炼出来的库，长期演化会遇到 UI 改版、操作系统更新这一类老问题：旧关键帧在新界面下还认不认得，状态卡的"可见线索"会不会因为重绘而失效。其三是与底层 LLM 的耦合度。生成器与门控都依赖大模型本身的图文理解能力，换底座可能整套都得重做评测。其四是结果的相对增益在小模型上看着很大，但起点也低（Qwen3-VL-8B 从 10.78% 升到 25.40%，意味着绝对成功率还是不到三成），离"可以放心交给它跑"的状态尚远。

## 一句话

把可复用技能从纯文本扩成"程序 + 状态卡 + 多视角关键帧"的多模态包，再用分支加载切断参考图与现场画面之间的视觉对抗，跨四个视觉智能体基准都见到一致的提升。
